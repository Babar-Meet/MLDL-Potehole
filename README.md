A Flask web application for detecting potholes in images and real-time video streams using a YOLOv8 deep learning model.

---

## Purpose

This project provides a web-based interface to upload road images and detect potholes using a trained YOLOv8 object detection model. It is intended for civil infrastructure inspection teams, municipal road maintenance departments, and researchers working on automated pothole detection.

The application supports both static image analysis and live video stream processing with real-time bounding box overlays.

---

## Features

- **Image upload and detection** — Upload road images (JPG, PNG, GIF, BMP, WebP) and receive detection results with blue bounding boxes and confidence scores.
- **Live video streaming** — Upload video files (MP4, AVI, MOV, MKV, FLV, WMV) and view a real-time MJPEG stream with per-frame pothole detection.
- **Real-time inference** — Each video frame is processed through the YOLOv8 model with configurable confidence threshold (default 25%).
- **Drag-and-drop upload** — Browser-based drag-and-drop zone with auto-submit on file selection.
- **Side-by-side comparison** — Original image and detection result displayed in a two-panel grid.
- **Full-screen image viewer** — Click any result image to open a modal with zoom-to-fit.
- **Build timestamp tracking** — Server exposes deployment/build timestamp via an API endpoint, displayed in the page footer.
- **Health check endpoint** — Simple `/health` endpoint for load balancers and monitoring.
- **Progress indication** — Spinner and status badge during image processing and video streaming.
- **Responsive dark UI** — Modern dark theme built with CSS custom properties, adapts to mobile viewports.
- **File conflict handling** — Duplicate filenames are automatically suffixed with an incrementing counter.
- **Video file loop** — Live video streams loop automatically when the end of file is reached.

## Architecture

### High-level design

The application follows a monolithic Flask architecture with a single Python process serving both static files and API endpoints. The YOLOv8 model is loaded once at startup and reused across all inference requests.

```
 User Browser                    Flask Server               File System
 ┌──────────┐    HTTP/Jinja     ┌──────────────┐           ┌──────────┐
 │          │ ◄──────────────► │  app.py      │  read/    │ uploads/ │
 │  index   │                  │  routes      │  write    │ results/ │
 │  .html   │                  │  inference   │ ◄───────► │          │
 │          │                  │  streaming   │           └──────────┘
 │  style   │                  │              │
 │  .css    │                  │  YOLO model  │           ┌──────────────────┐
 │          │                  │  (memory)    │  predict  │ pothole_yolo/    │
 └──────────┘                  │              │ ◄───────  │ train1/weights/  │
                               └──────────────┘           │ best.pt          │
                                                          └──────────────────┘
```

### Request flow — image detection

1. Browser loads `GET /` returning the rendered Jinja2 template.
2. User selects or drags an image file. The file input fires a change event.
3. JavaScript auto-submits `POST /upload` with `multipart/form-data`.
4. Flask validates the file extension, secures the filename, and writes to `uploads/`.
5. The YOLO model runs inference with `conf=0.25`, `save=True`. The saved result image lands in `results/output/`.
6. The template re-renders with `result_image`, `original_image`, `num_detections`, and side-by-side panels.

### Request flow — live video streaming

1. User switches to "Live" mode via the upload-type toggle.
2. User selects a video file; JavaScript calls `POST /start_live_video`.
3. Flask saves the video to `uploads/` and sets the global `is_processing_live = True`.
4. The browser sets an `<img>` `src` to `GET /video_feed/<filename>`.
5. Flask returns a `multipart/x-mixed-replace` response; a generator function reads frames via OpenCV `VideoCapture`, runs YOLO inference on each frame, draws bounding boxes, encodes to JPEG, and yields each frame.
6. When the video ends, OpenCV seeks back to frame 0 (loop).
7. User clicks "Stop Live Stream" → `POST /stop_live_video` → generator exits → stream ends.

### State management

- **No database.** All state is ephemeral and file-system based.
- A global boolean `is_processing_live` controls the video generator loop.
- The model object (`YOLO instance`) is a module-level singleton loaded once at import time.
- Uploaded and result files persist in `uploads/` and `results/` directories until manually cleaned.

### Error handling

- **413** — File too large (100 MB max for videos, enforced by Flask config).
- **400** — Missing file, empty filename, or invalid file extension.
- **500** — Model not loaded or any unhandled exception during inference.
- **404** — Resource not found (video file missing, invalid path).
- All errors return a JSON object `{"status": "error", "message": "...", "error": "ErrorCode"}` with appropriate HTTP status codes.

## APIs

### `GET /`
- **Purpose:** Serve the main web interface.
- **Response:** Rendered `index.html` Jinja2 template.
- **Authentication:** None.

### `GET /health`
- **Purpose:** Health check for load balancers and monitoring systems.
- **Response:** `{"status": "healthy"}`.
- **Authentication:** None.

### `GET /build-info`
- **Purpose:** Return the server build/deployment timestamp.
- **Response:** `{"build_timestamp": "...", "build_timestamp_iso": "...", "message": "Application build/deployment timestamp"}`.
- **Authentication:** None.

### `POST /upload`
- **Purpose:** Upload an image file and receive detection results.
- **Request:** `multipart/form-data` with field `file`.
- **Accepted types:** JPG, JPEG, PNG, GIF, BMP, WebP.
- **Response:** Rendered `index.html` template with `result_image`, `original_image`, `num_detections`.
- **Error responses:** 400 (invalid/missing file), 500 (model not loaded / inference error).
- **Authentication:** None.

### `POST /start_live_video`
- **Purpose:** Upload a video and start real-time detection streaming.
- **Request:** `multipart/form-data` with field `file`.
- **Accepted types:** MP4, AVI, MOV, MKV, FLV, WMV.
- **Response:** `{"status": "success", "filename": "...", "message": "..."}` on success, or `{"status": "error", ...}` on failure.
- **Error responses:** 400 (invalid file), 500 (save failure).
- **Authentication:** None.

### `POST /stop_live_video`
- **Purpose:** Stop an active live video stream.
- **Request:** Empty POST body.
- **Response:** `{"status": "success", "message": "Live video processing stopped"}`.
- **Authentication:** None.

### `GET /video_feed/<filename>`
- **Purpose:** Real-time MJPEG video stream with detection overlays.
- **Response:** `multipart/x-mixed-replace` stream of JPEG frames.
- **Error:** 404 if video file not found.
- **Authentication:** None.

### `GET /uploads/<path:filename>`
- **Purpose:** Serve uploaded files (images and videos).
- **Authentication:** None.

### `GET /results/<path:filename>`
- **Purpose:** Serve detection result images from `results/output/`.
- **Authentication:** None.

### `POST /live-video`
- **Purpose:** Returns a 400 error; this endpoint exists only as an API stub. The live video UI is handled entirely by the frontend.
- **Authentication:** None.

## How It Works

1. **Startup** — `app.py` initializes Flask, creates `uploads/` and `results/` directories, and loads the trained YOLOv8 model from `pothole_yolo/pothole_yolo/train1/weights/best.pt` into memory.
2. **Image detection** — A user uploads an image through the browser. The server saves the file, runs `model.predict()` with a confidence threshold of 0.25, and saves the annotated result image. The page re-renders showing the original image next to the result with blue bounding boxes and confidence labels.
3. **Live video streaming** — A user switches to Live mode and selects a video file. The server saves the file and starts a generator that reads frames via OpenCV. Each frame is passed through `model.predict()`, bounding boxes are drawn, the frame is JPEG-encoded, and yielded as part of a `multipart/x-mixed-replace` HTTP response. The browser displays the stream in an `<img>` tag and loops automatically when the video ends.
4. **Stopping** — The user clicks "Stop Live Stream," which sets `is_processing_live = False`, causing the generator to exit and the stream to end.
